## Executive Summary

### 한 줄 정의

**Onno(온노)**는 회의·미팅 상황에서 **실시간으로 질문·인사이트·맥락 설명을 제공**하고, 필요한 경우 CRM·PM·문서·메일 등 기존 시스템과 자연스럽게 연계해 **대화의 질과 실행력을 동시에 높여주는 실시간 대화 인텔리전스 플랫폼**입니다.

### 핵심 차별점

| 구분 | 기존 솔루션 (Otter, Fireflies) | Onno |
|-----|---------------------------|------|
| **초점** | 회의 '후' 전사·요약 | 회의 '중' 질문·인사이트 제공 |
| **가치** | "기록을 잘 남깁니다" | "대화를 잘하게 도와줍니다" |
| **시점** | After-the-fact | **During-the-fact** |
| **개인화** | 없음 (범용 요약) | **나만의 Onno** (학습·레벨업) |
| **Sherlocking 위험** | ⚠️ 높음 (MS/Zoom이 기본 제공) | ✅ 낮음 (실시간 인텔리전스 차별화) |

### 타겟 고객 (Initial Beachhead)

**"말로 판단하고, 말로 합의하는 직군"**
- VC/PE 투자 심사역
- Accelerator 프로그램 매니저
- Company Builder / Venture Studio PO
- 컨설턴트 (전략, 마케팅, 조직)
- 스타트업 창업가 (특히 B2B 세일즈, 투자 유치 과정)

### 1-2. 핵심 가치 제안 (Value Proposition)

#### 고객이 얻는 가치

**Before Onno** (기존 상황):
- VC 심사역: "질문을 '알지만', 회의 중엔 '듣기'에 90% 에너지를 쓰고 놓침"
- AC 매니저: "20개 팀 관리하는데, 매번 '이 팀 지난주 뭐 했지?' 떠올리기부터 시작"
- 창업가: "투자자가 뭘 궁금해할지 몰라서, 중요한 지표 설명 못하고 끝남"

**After Onno** (Onno 사용 후):
- VC: "회의 중 Onno가 '리텐션 코호트별 수치 물어보세요' 실시간 제안 → 즉시 질문"
- AC: "회의 시작하자마자 'A팀 지난주 마케팅 예산 고민' 자동 로드 → 맥락 파악 3초"
- 창업가: "투자자가 CAC 물어보기 전에 Onno가 '지금 CAC 설명하기 좋은 타이밍' 알림"

#### 3-Axis 핵심 기능

```
┌─────────────────────────────────────────────────────────────┐
│ Axis 1: Real-time Question Generation (실시간 질문 제안)    │
│ "지금 던져야 할 질문"                                          │
└─────────────────────────────────────────────────────────────┘
   입력: 실시간 STT 전사
   처리: 대화 단계 인식, 정보 누락 감지, 도메인 체크리스트 매칭
   출력: 우선순위별 질문 3~5개
         🔴 Critical (놓치면 안 되는 핵심)
         🟡 Important (추가 확인 권장)
         🟢 Follow-up (나중에 물어도 됨)

┌─────────────────────────────────────────────────────────────┐
│ Axis 2: Contextual Insight Layer (맥락 인사이트)             │
│ "이해를 돕는 배경 지식과 분석"                                  │
└─────────────────────────────────────────────────────────────┘
   입력: 발언 + 과거 대화 기록 + 도메인 벤치마크 DB
   처리: RAG 기반 맥락 검색, 지표 비교 분석, 용어 자동 설명
   출력:
   - "지난주 이 팀은 X 고민했음" 과거 맥락 카드
   - "CAC $50 = 동종 SaaS 상위 25%" 벤치마크 비교
   - "Net Retention이란..." 용어 설명 팝업

┌─────────────────────────────────────────────────────────────┐
│ Axis 3: Workflow Bridge (업무 워크플로우 연결)                │
│ "회의 결과 → 즉시 실행"                                        │
└─────────────────────────────────────────────────────────────┘
   입력: 회의 중 결정·액션 아이템 감지
   처리: NLU 기반 태스크 추출, 시스템 연동
   출력:
   - Notion에 "다음 주까지 재무 자료 요청" 태스크 자동 생성 (승인 필요)
   - Google Calendar에 "2주 후 Follow-up 미팅" 자동 예약
   - Slack에 "팀원에게 X 공유" 메시지 초안 작성

   ⚠️ Human-in-the-Loop: 자동 실행 전 반드시 사용자 승인
```

### 1-3. "나만의 Onno" - 개인화 시스템

#### 성장형 AI 파트너 컨셉

Onno는 단순한 도구가 아닌, **사용자와 함께 성장하는 대화 파트너**입니다.

**학습 메커니즘**:
1. **사용 패턴 학습**
   - 사용자가 선택한 질문 vs 무시한 질문 분석
   - 자주 사용하는 용어, 관심 도메인 파악
   - 회의 스타일 (공격적 vs 협력적, 데이터 중심 vs 전략 중심)

2. **도메인 레벨 시스템**
   ```
   투자 심사   ⭐⭐⭐⭐☆ Lv.4 (Expert)
   마케팅 전략 ⭐⭐⭐☆☆ Lv.3 (Advanced)
   제품 개발   ⭐⭐☆☆☆ Lv.2 (Intermediate)
   조직 문화   ⭐☆☆☆☆ Lv.1 (Beginner)
   ```
   - 특정 도메인 회의를 많이 할수록 레벨 상승
   - 높은 레벨 = 더 정교한 질문, 더 깊은 인사이트

3. **개인화된 인터페이스**
   - 캐릭터/아바타 선택 가능 (선택사항)
   - 질문 톤 조정 (공손함 vs 직설적, 간결 vs 상세)
   - 알림 강도 설정 (적극적 제안 vs 수동적 대기)

**전환 비용(Switching Cost) 확보**:
- 3개월 사용 후: Onno가 나의 대화 스타일, 관심 도메인 완전히 학습
- 타 서비스로 전환 시: 처음부터 다시 학습해야 함 → 높은 전환 비용
- → **장기 Retention 및 브랜드 충성도 확보**

## 2. 문제 정의 (Problem Statement)

### 2-1. 시장 배경: Sherlocking의 위협

**현재 회의 솔루션 시장의 딜레마**:

```
[2022-2023: Act 1 - 요약 툴 범람]
Otter, Fireflies, Tactiq, Fathom 등 수십 개 서비스 출시
→ "회의 자동 전사·요약" 가치 제안

[2024: 빅테크의 역습]
→ Microsoft Teams Copilot (무료/번들)
→ Zoom AI Companion (무료/번들)
→ Google Meet Gemini (무료/번들)

[결과: Sherlocking]
"전사·요약"만으로는 생존 불가
→ 고객이 "굳이 별도 서비스 쓸 이유 없음"
```

**Onno의 대응 전략**:
- ❌ 전사·요약 경쟁 회피 (빅테크와 싸우지 않음)
- ✅ **실시간 대화 인텔리전스**라는 새로운 카테고리 창출
- ✅ MS/Zoom이 쉽게 따라올 수 없는 **Vertical 깊이** (도메인 특화)

### 2-2. 고객 Pain Point: "인지적 병목"

#### 근본 문제: 인간은 실시간 대화 중 4가지를 동시에 못함

```
┌─────────────────────────────────────────────────────────┐
│  실시간 대화 중 필요한 4가지 인지 작업                      │
└─────────────────────────────────────────────────────────┘

1. 듣기 (Listening)          → 60% 에너지 소모
   상대방 발언 이해, 의도 파악

2. 분석 (Analyzing)          → 20% 에너지 소모
   논리적 허점, 누락된 정보, 리스크 감지

3. 기억 (Recalling)          → 10% 에너지 소모
   과거 대화 맥락, 도메인 지식, 벤치마크 상기

4. 생성 (Generating)         → 10% 에너지 소모
   다음 질문 구성, 후속 조치 계획

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
결과: 90%를 "듣기"에 쓰고, 나머지에 10%만 남음
→ "질문을 '알지만' '던지지' 못하는 상황" 발생
```

#### 페르소나별 Pain Point (15개 발견)

**공통 Pain (모든 사용자)**:
1. **Question Timing Gap**: 질문을 알지만 적시에 던지지 못함
2. **Surface Acceptance**: 그럴듯한 답변에 속아 넘어감
3. **Checklist Dormancy**: 체크리스트가 회의 중 작동 안 함
4. **Follow-up Weakness**: 회의 후 후속 액션 누락
5. **Context Recall Failure**: 이전 회의 맥락 기억 못함

**VC 특화 Pain**:
6. **Stage Blindness**: Seed/Series A/B 단계별 필수 질문 놓침

**Accelerator 특화 Pain**:
7. **Teaching Overload**: 같은 개념 매 팀에 반복 설명
8. **Benchmark Lag**: 실시간 벤치마크 비교 불가
9. **Cohort Tracking Gap**: 여러 팀 동시 관리 시 맥락 손실

**Company Builder 특화 Pain**:
10. **Echo Chamber**: 내부 편향 검증 없이 진행
11. **Reporting Data Gap**: 회의 → 보고서 변환 수작업
12. **Interview Follow-up Weakness**: 사용자 인터뷰 후 검증 누락

**전체 Pain 요약**:
- "나는 질문을 '알고' 있지만, '던지지' 못한다" (VC 5년차)
- "20개 팀 관리하는데, 매번 '이 팀 지난주 뭐 했지?'부터 시작" (AC 매니저)
- "사용자 인터뷰 10건 했는데, 가설 검증하려면 녹취록 다시 들어야 함. 안 함" (CB PO)

### 2-3. 기존 솔루션의 한계

| 솔루션 | 해결하는 것 | 해결 못하는 것 | Onno의 차별점 |
|--------|-----------|--------------|-------------|
| **Fireflies/Otter** | 사후 '기억' (녹취록 검색) | 실시간 '분석·질문 생성' | ✅ 회의 중 질문 제안 |
| **Gong** | 사후 '코칭' (Sales 분석) | 실시간 '질문 제안' | ✅ During-the-fact |
| **Fellow** | 사전 '준비' (아젠다) | 실시간 '적응' (맥락 변화) | ✅ 맥락 인식 질문 |
| **Notion/Airtable** | 사전 '체크리스트' | 회의 중 '활성화' | ✅ 체크리스트 자동 트리거 |

**Onno의 유일무이한 가치**:
→ **회의 '중' 실시간 질문·인사이트 제공** (During-the-fact Intelligence)

### 3-2. 핵심 기능 플로우

#### 회의 전 (Pre-meeting)

```
[사용자 액션]
1. Onno 웹앱에서 "새 회의 시작" 클릭
2. 회의 유형 선택 (투자 심사 / 멘토링 / 세일즈 / 제품 리뷰 등)
3. (선택) 참석자 선택 → 과거 대화 자동 로드

[Onno 자동 처리]
→ 과거 회의 기록 검색 (RAG)
   "지난번 A팀과의 미팅: 마케팅 예산 고민 중"
→ 투자 단계 자동 인식
   "Series A 심사 → 필수 질문 20개 활성화"
→ 체크리스트 로드
   사용자가 Notion/Airtable에 저장한 템플릿 자동 불러오기
```


## 4. MVP 개발 계획

### 4-1. MVP 철학: 범용 접근

**Why 범용 MVP?**

기존 계획은 "VC/AC/CB 버티컬 특화"였으나, MVP 단계에서는 다음 이유로 **범용** 접근:

1. **작동 검증 우선**: 실시간 질문 생성 엔진이 "작동하는지" 먼저 확인
2. **빠른 피드백 루프**: 다양한 도메인에서 테스트 → 어디서 가장 반응 좋은지 발견
3. **개발 속도**: 도메인별 커스터마이징 없이, 핵심 엔진에만 집중
4. **회사별 온보딩은 Phase 2**: 검증 후 유료화 시점에 도입

**범용 MVP 기능**:
- ✅ 실시간 STT + 전사
- ✅ 범용 질문 생성 (GPT-4o Prompt Engineering)
- ✅ 기본 용어 설명 (50개 비즈니스 용어)
- ✅ 액션 아이템 자동 추출
- ✅ Notion 연동 (가장 범용적인 도구)
- ❌ 도메인별 체크리스트 (Phase 2)

## 7. 성공 지표 및 Pivot 기준

### 7-1. North Star Metric

**Breakdown**:

- Impact: 질문 사용률 (실제로 던진 비율)

### 7-2. PMF 검증 지표 (6개월 후)

| 지표 | 목표 | 측정 방법 | Benchmark |
|-----|------|----------|-----------|

### 7-3. Pivot 트리거

#### Scenario 1: Retention 붕괴 (<30%)
- **신호**: 처음엔 쓰다가 4주 후 이탈
- **원인 가설**: 실시간 제안이 방해됨 (Intrusive)
- **Pivot**: During → After (회의 후 코칭 강화)

#### Scenario 2: 질문 사용률 폭락 (<20%)
- **신호**: 질문 보지만 안 던짐
- **원인**: 질문 품질 낮음, 타이밍 안 맞음
- **Pivot**: 질문 생성 포기 → 체크리스트 자동 활성화만

#### Scenario 3: 유료 전환 제로
- **신호**: "좋긴 한데 돈 낼 정도는 아니에요"
- **원인**: Value proposition 약함
- **Pivot**: B2C → B2B2C (AC가 포트폴리오에 제공)

#### Scenario 4: Sherlocking 현실화
- **신호**: MS/Zoom이 비슷한 기능 출시
- **원인**: Platform risk
- **Pivot**: 더 깊은 Vertical (Legal AI, Healthcare)

### 8-2. 방어 가능성 (Defensibility)

#### 1. 데이터 모트 (Data Moat)
```
사용자가 Onno를 3개월 사용
→ 과거 회의 50건 누적
→ 도메인 지식 학습 (투자 심사 Lv.5)
→ 타 서비스로 전환 시 처음부터 다시 학습
→ 높은 전환 비용 (Switching Cost)
```

#### 2. 네트워크 효과 (Network Effect)
```
[Phase 3: Marketplace]
사용자가 체크리스트·질문 템플릿 제작
→ 다른 사용자가 구매·사용
→ 더 많은 사용자 유입
→ 더 많은 템플릿 생산
→ Flywheel
```

#### 3. Vertical 깊이 (Domain Expertise)
```
VC 투자 심사 워크플로우 완전 이해
→ QuotaBook, VCworks 등 한국 VC 도구 통합
→ 한국 VC만의 Pain (HWP, PIPA) 해결
→ 해외 솔루션이 쉽게 따라올 수 없음
```

#### 4. 브랜드 충성도 (Brand Loyalty)
```
"나만의 Onno" 개인화 경험
→ 캐릭터, 레벨, 학습 히스토리
→ 감정적 애착 형성
→ "Onno 없이 회의 못해요" 상태
```

### 8-3. 왜 MS/Zoom이 쉽게 못 따라올까?

1. **Horizontal vs Vertical**:
   - MS/Zoom: 모든 산업, 모든 회의 유형 대응 (범용)
   - Onno: VC/투자 심사만 극도로 깊게 (특화)
   - → MS가 "VC 전용 질문 엔진" 만들 리소스 투입할 가능성 낮음

2. **Platform Neutrality**:
   - Onno: Zoom + Meet + Teams 모두 지원
   - MS: Teams만, Zoom: Zoom만
   - → 고객은 멀티 플랫폼 솔루션 선호

3. **Agility**:
   - 빅테크: 의사결정 느림, 실험 어려움
   - 스타트업: 피드백 → 배포 1주일
   - → 빠른 진화로 격차 유지

### B. 기술적 도전 과제 및 해결 방안

#### 도전 1: 실시간 Latency
**문제**: STT → LLM 추론 → 질문 생성 = 5~10초 걸릴 수 있음
**해결**:
- STT 스트리밍 (Whisper API 대신 Deepgram)
- LLM Caching (반복 패턴 캐싱)
- 목표: 발언 종료 후 2초 내 질문 제안

#### 도전 2: 질문 품질 (Hallucination)
**문제**: LLM이 엉뚱한 질문 생성 (Hallucination)
**해결**:
- Fine-tuning (VC 실제 질문 500개 학습)
- Rule-based 검증 (도메인 체크리스트와 대조)
- User Feedback Loop (사용/무시 데이터로 재학습)

#### 도전 3: 한국어 정확도
**문제**: 한국어 STT·NLU 정확도 낮을 수 있음
**해결**:
- Whisper Large v3 (한국어 최고 성능)
- Claude 3.5 (한국어 이해 우수)
- 한국어 전문 벤치마크 테스트 (사투리, 전문 용어)

#### 도전 4: 개인정보 보호 (PIPA)
**문제**: 회의 녹음 = 개인정보, 민감 정보 포함
**해결**:
- E2E 암호화 (S3 저장 시 AES-256)
- 사용자 명시적 동의 (녹음 시작 전 팝업)
- 자동 삭제 옵션 (회의 종료 후 7일 자동 삭제)
- On-premise 옵션 (Enterprise, Phase 3)
